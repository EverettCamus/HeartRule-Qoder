# 开发流程规范 (Development Process Specification)

## 概述

本文档定义HeartRule AI咨询引擎项目的开发流程规范，基于Scrum敏捷开发框架，结合OpenSpec文档驱动开发方法。

## 1. Scrum开发流程

### 1.1 Sprint周期

- **周期长度**: 2周（10个工作日）
- **节奏**: 固定节奏，周一启动，周五结束
- **例外处理**: 节假日顺延，提前通知团队

### 1.2 Sprint规划会议 (Sprint Planning)

- **时间**: Sprint开始前最后一天，2小时
- **参与者**: 产品负责人、Scrum Master、开发团队
- **输入**: 产品待办列表 (Product Backlog)
- **输出**: Sprint待办列表 (Sprint Backlog)
- **议程**:
  1. 产品负责人讲解最高优先级需求
  2. 团队评估工作量（故事点）
  3. 确定Sprint目标
  4. 创建Sprint待办列表

### 1.3 每日站会 (Daily Scrum)

- **时间**: 工作日早上9:30，15分钟
- **格式**: 每人回答三个问题：
  1. 昨天完成了什么？
  2. 今天计划做什么？
  3. 遇到什么障碍？
- **规则**: 只讨论进展和障碍，不深入技术讨论

### 1.4 Sprint评审会议 (Sprint Review)

- **时间**: Sprint最后一天下午，1.5小时
- **参与者**: 产品负责人、Scrum Master、开发团队、利益相关者
- **目的**: 演示完成的功能，收集反馈
- **输出**: 产品待办列表更新

### 1.5 Sprint回顾会议 (Sprint Retrospective)

- **时间**: Sprint评审会议后，1小时
- **参与者**: Scrum Master、开发团队
- **目的**: 改进开发流程
- **格式**: 开始/停止/继续 (Start/Stop/Continue)
- **输出**: 改进项行动计划

## 2. OpenSpec工作流

### 2.1 文档驱动开发流程

```
需求分析 → OpenSpec提案 → 团队评审 → 实施 → 验证 → 归档
```

### 2.2 OpenSpec提案流程

1. **需求分析**: 分析用户需求，确定问题域
2. **创建提案**: 使用`/opsx-propose`命令创建OpenSpec提案
3. **生成工件**: 自动生成设计文档、任务列表、测试计划
4. **团队评审**: 团队评审提案，提出修改意见
5. **批准实施**: 产品负责人批准后开始实施

### 2.3 实施流程

1. **任务分解**: 将OpenSpec提案分解为具体任务
2. **创建分支**: 为每个任务创建特性分支
3. **代码实现**: 遵循TDD（测试驱动开发）原则
4. **代码审查**: 提交Pull Request，进行代码审查
5. **合并部署**: 通过审查后合并到主分支

### 2.4 归档流程

1. **完成验证**: 确保所有任务完成，测试通过
2. **更新文档**: 更新相关文档，记录变更
3. **归档变更**: 使用`/opsx-archive`命令归档OpenSpec变更
4. **知识沉淀**: 将经验沉淀到团队知识库

## 3. 代码质量流程

### 3.1 代码审查规范

- **审查范围**: 所有代码变更必须经过审查
- **审查时间**: 提交PR后24小时内完成审查
- **审查重点**:
  1. 代码正确性
  2. 代码可读性
  3. 测试覆盖率
  4. 性能影响
  5. 安全考虑

### 3.2 测试策略

- **单元测试**: 覆盖率目标80%+
- **集成测试**: 关键路径100%覆盖
- **端到端测试**: 核心用户流程覆盖
- **测试执行**: 每次提交自动运行测试

### 3.3 代码风格

- **TypeScript**: 遵循项目ESLint配置
- **命名规范**: 遵循项目命名约定
- **注释要求**: 公共API必须有JSDoc注释
- **提交信息**: 遵循Conventional Commits规范

## 4. 部署流程

### 4.1 开发环境部署

- **自动化**: 每次合并到主分支自动部署到开发环境
- **验证**: 部署后自动运行冒烟测试
- **通知**: 部署结果通知团队

### 4.2 测试环境部署

- **手动触发**: 需要手动批准
- **完整测试**: 运行所有测试套件
- **性能测试**: 关键路径性能验证

### 4.3 生产环境部署

- **发布时间**: 每周四下午（非紧急情况）
- **变更窗口**: 2小时变更窗口
- **回滚计划**: 必须有明确的回滚方案
- **监控**: 部署后密切监控系统指标

## 5. 团队协作规范

### 5.1 沟通渠道

- **日常沟通**: Slack/Teams即时消息
- **会议沟通**: Zoom/Teams视频会议
- **文档沟通**: Confluence/OpenSpec文档
- **代码沟通**: GitHub Issues/Pull Requests

### 5.2 知识管理

- **设计决策**: 记录在OpenSpec设计文档中
- **技术债务**: 记录在技术债务跟踪表中
- **经验教训**: 记录在Sprint回顾会议纪要中
- **最佳实践**: 沉淀到团队知识库

### 5.3 新人入职

- **入职清单**: 标准化的入职清单
- **导师制度**: 指定导师协助新人
- **培训材料**: 完整的培训文档和视频
- **实践项目**: 小规模实践项目快速上手

## 6. 流程改进机制

### 6.1 定期评审

- **频率**: 每季度评审一次开发流程
- **参与者**: 所有团队成员
- **目标**: 识别流程瓶颈，提出改进建议

### 6.2 指标监控

- **交付速度**: 故事点完成率
- **代码质量**: 测试覆盖率、代码审查通过率
- **团队满意度**: 定期团队满意度调查
- **客户满意度**: 客户反馈收集和分析

### 6.3 持续改进

- **小步快跑**: 每次改进一个小方面
- **实验精神**: 尝试新方法，评估效果
- **数据驱动**: 基于数据做决策
- **团队共识**: 改进需要团队共识
