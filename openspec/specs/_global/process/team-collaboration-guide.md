# 团队协作指南 (Team Collaboration Guide)

## 概述

本文档定义HeartRule AI咨询引擎项目的团队协作规范，旨在建立高效、透明、健康的团队协作环境。

## 1. 团队价值观

### 1.1 核心原则

- **用户第一**: 所有决策以用户价值为导向
- **质量至上**: 不妥协代码质量和系统稳定性
- **持续学习**: 鼓励知识分享和技术探索
- **相互尊重**: 尊重不同观点，建设性沟通
- **结果导向**: 关注交付价值，而非仅仅完成任务

### 1.2 行为准则

1. **守时**: 准时参加会议，按时交付承诺
2. **透明**: 及时沟通进展、问题和风险
3. **负责**: 对自己负责的工作承担到底
4. **协作**: 主动帮助团队成员，共同解决问题
5. **改进**: 持续反思和改进工作方式

## 2. 沟通规范

### 2.1 沟通渠道

| 场景     | 推荐渠道            | 响应时间 | 备注                   |
| -------- | ------------------- | -------- | ---------------------- |
| 紧急问题 | 电话/即时消息       | 立即     | 影响系统运行的关键问题 |
| 技术讨论 | Slack技术频道       | 4小时内  | 非紧急技术问题         |
| 任务协调 | GitHub Issues/PR    | 24小时内 | 代码相关协作           |
| 文档协作 | OpenSpec/Confluence | 48小时内 | 设计文档评审           |
| 团队公告 | 团队邮件列表        | 按需     | 重要团队通知           |

### 2.2 会议规范

- **会前准备**: 提前发送议程和材料
- **准时开始**: 尊重所有人的时间
- **明确角色**: 指定主持人、记录员
- **聚焦议题**: 不偏离会议主题
- **会后跟进**: 发送会议纪要和行动项

### 2.3 异步沟通

- **清晰主题**: 邮件/消息主题明确说明内容
- **结构清晰**: 使用列表、标题等提高可读性
- **一次说完**: 尽量一次性提供完整信息
- **明确期望**: 明确说明需要对方做什么
- **及时回复**: 在承诺时间内回复

## 3. 决策流程

### 3.1 决策类型

| 决策类型 | 决策者       | 咨询对象           | 文档要求     |
| -------- | ------------ | ------------------ | ------------ |
| 技术决策 | 技术负责人   | 相关开发人员       | 技术设计文档 |
| 产品决策 | 产品负责人   | 用户代表、开发团队 | 产品需求文档 |
| 架构决策 | 架构师       | 全体技术团队       | 架构设计文档 |
| 流程决策 | Scrum Master | 全体团队成员       | 流程改进提案 |

### 3.2 决策流程

```
问题识别 → 信息收集 → 方案提出 → 方案评审 → 决策执行 → 效果评估
```

### 3.3 共识建立

- **充分讨论**: 给所有人表达意见的机会
- **数据驱动**: 基于事实和数据做决策
- **明确标准**: 建立清晰的决策标准
- **记录决策**: 记录决策理由和依据
- **接受分歧**: 允许合理分歧，但决策后统一执行

## 4. 冲突解决

### 4.1 冲突预防

- **明确职责**: 清晰的角色和职责定义
- **透明沟通**: 及时沟通期望和问题
- **定期反馈**: 定期的一对一反馈会议
- **团队建设**: 定期的团队建设活动

### 4.2 冲突解决步骤

1. **直接沟通**: 当事人直接沟通解决
2. **团队调解**: 寻求团队领导或同事帮助
3. **正式调解**: 人力资源或上级介入
4. **升级处理**: 严重冲突按公司政策处理

### 4.3 建设性冲突

- **对事不对人**: 聚焦问题，不攻击个人
- **寻求共赢**: 寻找双方都能接受的方案
- **保持尊重**: 即使不同意，也保持尊重
- **向前看**: 解决后不翻旧账

## 5. 知识管理

### 5.1 知识沉淀

- **设计决策**: 记录在OpenSpec设计文档中
- **技术方案**: 记录在技术方案文档中
- **经验教训**: 记录在Sprint回顾会议纪要中
- **最佳实践**: 沉淀到团队知识库

### 5.2 知识分享

- **技术分享会**: 每月一次技术分享
- **代码审查**: 通过代码审查分享经验
- **结对编程**: 定期结对编程活动
- **文档协作**: 共同编写和维护文档

### 5.3 新人培养

- **导师制度**: 为新成员指定导师
- **入职清单**: 标准化的入职流程
- **培训计划**: 个性化的培训计划
- **实践项目**: 从简单到复杂的实践项目

## 6. 工作习惯

### 6.1 代码协作

- **分支策略**: 遵循Git Flow分支策略
- **提交规范**: 遵循Conventional Commits规范
- **代码审查**: 所有代码必须经过审查
- **测试先行**: 遵循测试驱动开发原则

### 6.2 文档协作

- **文档即代码**: 文档与代码同等重要
- **及时更新**: 代码变更时同步更新文档
- **版本控制**: 文档纳入版本控制系统
- **定期评审**: 定期评审和更新文档

### 6.3 时间管理

- **专注时间**: 每天保留2-4小时不被打扰的专注时间
- **任务分解**: 大任务分解为可管理的小任务
- **优先级管理**: 使用四象限法管理任务优先级
- **定期回顾**: 每周回顾任务完成情况

## 7. 团队健康

### 7.1 工作生活平衡

- **合理工时**: 避免长期加班
- **灵活工作**: 支持合理的灵活工作安排
- **休假制度**: 鼓励使用年假，充分休息
- **健康关怀**: 关注团队成员身心健康

### 7.2 团队建设

- **定期活动**: 每月一次团队建设活动
- **庆祝成功**: 庆祝项目里程碑和团队成就
- **开放交流**: 创造开放、安全的交流环境
- **互相支持**: 团队成员互相支持和帮助

### 7.3 反馈文化

- **定期反馈**: 定期的一对一反馈会议
- **360度反馈**: 全面的反馈机制
- **建设性反馈**: 聚焦行为改进，而非个人批评
- **接受反馈**: 以开放心态接受反馈

## 8. 工具使用

### 8.1 协作工具

| 工具类型 | 推荐工具            | 主要用途               |
| -------- | ------------------- | ---------------------- |
| 代码管理 | GitHub              | 代码版本控制、代码审查 |
| 项目管理 | Jira/Trello         | 任务跟踪、Sprint管理   |
| 文档协作 | OpenSpec/Confluence | 设计文档、知识管理     |
| 即时通讯 | Slack/Teams         | 日常沟通、技术讨论     |
| 视频会议 | Zoom/Teams          | 远程会议、团队沟通     |

### 8.2 工具使用规范

- **统一工具**: 团队使用统一的工具套件
- **规范使用**: 遵循工具使用规范
- **定期培训**: 定期进行工具使用培训
- **反馈改进**: 收集工具使用反馈，持续改进
