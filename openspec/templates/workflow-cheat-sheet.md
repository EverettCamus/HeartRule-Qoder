# OpenSpec 工作流程速查表

## 概述

本文档提供OpenSpec工作流程的快速参考，帮助团队高效使用OpenSpec工具进行文档驱动开发。

## 1. 核心命令速查

### 1.1 提案命令

```bash
# 创建新提案（完整流程）
/opsx-propose "提案标题" --description "提案描述"

# 示例
/opsx-propose "实现用户认证系统" --description "为HeartRule添加JWT-based用户认证"
```

### 1.2 探索命令

```bash
# 进入探索模式（思考、调研、澄清需求）
/opsx-explore "探索主题"

# 示例
/opsx-explore "如何设计可扩展的会话存储系统"
```

### 1.3 实施命令

```bash
# 开始实施OpenSpec提案
/opsx-apply <change-id>

# 示例
/opsx-apply change_abc123
```

### 1.4 归档命令

```bash
# 归档已完成变更
/opsx-archive <change-id>

# 示例
/opsx-archive change_abc123
```

## 2. 工作流程速查

### 2.1 完整工作流

```
需求分析 → /opsx-propose → 团队评审 → /opsx-apply → 实施 → 验证 → /opsx-archive
```

### 2.2 快速启动

1. **明确需求**: 确定要解决的问题
2. **创建提案**: `/opsx-propose "标题" --description "描述"`
3. **评审完善**: 团队评审，补充细节
4. **开始实施**: `/opsx-apply <change-id>`
5. **完成任务**: 完成所有任务项
6. **归档变更**: `/opsx-archive <change-id>`

## 3. 提案结构模板

### 3.1 提案标题格式

```
[类型] 功能描述
```

**类型示例**:

- `[FEAT]`: 新功能
- `[FIX]`: Bug修复
- `[REFACTOR]`: 重构
- `[DOCS]`: 文档更新
- `[PERF]`: 性能优化

### 3.2 提案内容结构

```markdown
## 问题描述

[清晰描述要解决的问题]

## 目标

[列出具体、可衡量的目标]

## 解决方案概述

[简要描述解决方案]

## 实施计划

1. [任务1]
2. [任务2]
3. [任务3]

## 成功标准

- [ ] 标准1
- [ ] 标准2
- [ ] 标准3

## 风险与缓解

- **风险1**: [描述] → **缓解**: [措施]
- **风险2**: [描述] → **缓解**: [措施]
```

## 4. 目录结构速查

### 4.1 OpenSpec根目录

```
openspec/
├── config.yaml          # 配置文件
├── changes/            # 变更记录
├── specs/              # 规范文档
│   ├── product/        # 产品规格
│   ├── domain/         # 领域设计
│   ├── architecture/   # 架构设计
│   ├── research/       # 技术研究
│   └── _global/        # 全局规范
├── schemas/            # 模式定义
├── scripts/            # 工具脚本
└── templates/          # 文档模板
```

### 4.2 文档分类指南

| 文档类型 | 存放目录                            | 语言 | 示例                   |
| -------- | ----------------------------------- | ---- | ---------------------- |
| 产品需求 | `specs/product/`                    | 中文 | 用户故事、产品待办列表 |
| 战略设计 | `specs/domain/strategic/`           | 中文 | 限界上下文、核心域     |
| 战术设计 | `specs/domain/tactical/`            | 英文 | 实体、值对象、领域服务 |
| 架构设计 | `specs/architecture/`               | 英文 | 系统架构、组件设计     |
| AI研究   | `specs/research/ai-implementation/` | 中文 | AI实现机制研究         |
| 技术研究 | `specs/research/technical/`         | 英文 | 技术方案研究           |
| 流程规范 | `specs/_global/process/`            | 中文 | 开发流程、团队协作     |

## 5. 语言策略速查

### 5.1 语言使用规则

- **中文文档**: 产品需求、用户故事、战略设计、AI研究
- **英文文档**: 架构设计、战术设计、技术研究、API文档
- **混合文档**: 不允许，必须按类型分离

### 5.2 语言验证

```bash
# 运行语言验证脚本
./openspec/templates/language-validation-script.sh

# 检查特定目录
./openspec/templates/language-validation-script.sh specs/domain/strategic/
```

## 6. 迁移工作流速查

### 6.1 文档迁移步骤

1. **评估**: 检查文档是否符合迁移条件
2. **分类**: 确定目标目录和语言
3. **迁移**: 复制到目标位置，更新格式
4. **验证**: 运行验证脚本
5. **更新引用**: 更新项目中的文档引用
6. **记录**: 更新迁移跟踪文档

### 6.2 迁移条件

**迁移**:

- 核心产品需求
- 关键架构决策
- 重要DDD设计
- 重要研究成果
- 活跃开发文档

**不迁移**:

- 临时调试文档
- 已弃用功能文档
- 重复内容
- 特定bug修复记录
- 个人笔记

## 8. 常见问题速查

### 8.1 提案相关

**Q: 提案被拒绝怎么办？**
A: 根据反馈修改后重新提交，或创建新的探索会话澄清需求。

**Q: 提案太大怎么办？**
A: 拆分为多个小提案，每个聚焦一个具体功能。

**Q: 如何确定提案优先级？**
A: 基于用户价值、技术风险、实施成本综合评估。

### 8.2 实施相关

**Q: 任务卡住了怎么办？**
A: 使用`/opsx-explore`进入探索模式，或寻求团队帮助。

**Q: 需求变更怎么办？**
A: 创建新的提案，或修改现有提案（需团队评审）。

**Q: 如何跟踪进度？**
A: 使用OpenSpec内置的任务跟踪功能，定期更新状态。

### 8.3 工具相关

**Q: 命令不工作怎么办？**
A: 检查OpenSpec CLI是否安装，或联系系统管理员。

**Q: 文件找不到怎么办？**
A: 检查目录结构，确保文件在正确位置。

**Q: 验证失败怎么办？**
A: 根据错误信息修正问题，或寻求技术帮助。

## 9. 最佳实践

### 9.1 提案最佳实践

- **小步快跑**: 每个提案聚焦一个具体功能
- **明确目标**: 定义清晰、可衡量的成功标准
- **充分调研**: 实施前充分调研技术方案
- **团队协作**: 提案创建后及时寻求反馈

### 9.2 文档最佳实践

- **及时更新**: 代码变更时同步更新文档
- **版本控制**: 文档与代码一起版本控制
- **定期评审**: 定期评审和更新文档
- **统一格式**: 遵循统一的文档格式规范

### 9.3 协作最佳实践

- **透明沟通**: 及时沟通进展和问题
- **主动分享**: 主动分享知识和经验
- **互相评审**: 互相评审提案和代码
- **持续改进**: 定期回顾和改进工作流程

## 10. 紧急情况处理

### 10.1 生产问题

1. **立即响应**: 使用紧急通道通知团队
2. **快速修复**: 创建紧急修复提案
3. **实施验证**: 快速实施和验证
4. **事后分析**: 问题解决后进行根本原因分析

### 10.2 工具故障

1. **备用方案**: 使用传统文档和代码管理工具
2. **问题报告**: 立即报告给工具维护团队
3. **恢复计划**: 制定数据恢复和工具恢复计划
4. **预防措施**: 分析原因，制定预防措施
