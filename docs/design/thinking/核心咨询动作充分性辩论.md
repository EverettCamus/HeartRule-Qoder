# 核心咨询动作充分性辩论

## 辩题
**三个咨询动作单元（ai_say、ai_ask、ai_think）是否足以模拟完整的面对面咨询会谈？**

---

## 第一轮：基础能力边界探讨

### 正方论述
从信息流动的本质看，咨询会谈可以抽象为三种基本操作：

1. **信息输出**（ai_say）：咨询师向来访者传递知识、反馈、解释
2. **信息收集**（ai_ask）：咨询师引导来访者表达、澄清、探索
3. **信息处理**（ai_think）：咨询师对收集信息进行整理、分析、推理

任何复杂的咨询技术都可以分解为这三种操作的组合序列。例如：
- **认知重构** = ai_ask（收集认知）+ ai_think（分析扭曲）+ ai_say（提供新视角）
- **情绪聚焦** = ai_ask（引导情绪表达）+ ai_think（识别核心情绪）+ ai_say（确认与命名）
- **苏格拉底提问** = 多轮ai_ask + ai_think（引导逻辑链条）

从图灵完备性的角度类比，这三个原子操作构成了咨询会谈的"指令集"。

### 反方论述
这种简化忽略了咨询会谈的多个关键维度：

1. **非语言信息处理缺失**
   - 来访者的沉默、犹豫、回避本身携带信息
   - ai_ask只能"提问"，无法表达"等待"这一咨询师的关键行为
   - 缺少"ai_observe"（观察）或"ai_silent"（沉默）来处理非主动言语信息

2. **情感调节功能缺位**
   - 咨询师的共情不仅是"说出理解"（ai_say），更包含情感同频
   - 当来访者崩溃时，咨询师需要"ai_contain"（情感容纳）而非继续提问
   - 情感节奏的把控不等于信息的输入输出

3. **关系动力学处理不足**
   - 移情/反移情的觉察和处理需要"ai_meta_reflect"（元反思）
   - 咨询关系本身成为干预工具时，单纯的信息流动模型失效

4. **危机干预的行动性**
   - 当出现自杀风险时，咨询师需要"ai_intervene"（主动干预）
   - 这不是信息输出，而是关系性的控制边界设定

## 第二轮：操作分解可行性质疑

### 正方反驳
反方提出的"缺失功能"实际可以通过现有三个动作的参数化实现：

1. **沉默与等待**
   ```yaml
   - action: ai_say
     content_template: ""  # 空输出
     duration: 3000        # 停顿时长
     purpose: "give_space_for_reflection"
   ```

2. **情感容纳**
   ```yaml
   - action: ai_say
     content_template: "我能感受到这对你来说很艰难"
     tone: "warm_validating"  # 通过参数控制情感色彩
   - action: ai_think
     focus: "assess_emotional_tolerance"
   ```

3. **元反思**
   ```yaml
   - action: ai_think
     scope: "therapist_countertransference"
     output_variables: ["my_emotional_reaction"]
   - action: ai_say
     content_template: "我注意到我们的互动中..."
   ```

4. **危机干预**
   ```yaml
   - action: ai_ask
     content_template: "你现在有伤害自己的具体计划吗？"
     urgency: "crisis"
     required: true
   - action: ai_say
     content_template: "我需要确保你的安全，让我们一起..."
     directive: true
   ```

关键在于：**通过配置参数扩展动作的语义丰富度，而非增加动作类型**。

### 反方深入质疑
参数化方案暴露了更深层的问题：

1. **语义膨胀导致抽象失效**
   - 当ai_say需要通过`tone`、`directive`、`duration`等几十个参数来区分"共情"、"解释"、"沉默"、"干预"时，"say"这个抽象已经失去意义
   - 这相当于用"移动"动作加100个参数来模拟"跑步"、"跳跃"、"攀爬"——形式上统一，实质上混乱

2. **时序控制的复杂性**
   - 咨询师的"等待"不是简单的3秒停顿，而是**对来访者状态的持续监测直到某条件满足**
   - 需要"ai_wait_until"这种带条件的阻塞动作
   - 例：等待来访者完成哭泣、等待情绪强度降至可谈话水平

3. **双向互动的不对称性**
   - ai_ask假设"咨询师主动→来访者被动响应"的单向流动
   - 真实会谈中存在"来访者主动提问"、"来访者挑战咨询师"的反向流动
   - 需要"ai_respond"（响应来访者的主动询问）这种被动触发的动作

4. **决策点的显式化需求**
   ```
   真实场景：来访者沉默30秒后
   - 如果低头回避 → 温和探询
   - 如果泪眼凝视 → 继续等待
   - 如果愤怒盯视 → 命名关系张力
   ```
   这需要"ai_branch"（分支决策）而非线性的ask-think-say序列

## 第三轮：最小完备集的边界讨论

### 正方策略调整
承认需要增加**流程控制原语**，但信息处理核心仍保持三元结构：

**修正后的最小完备集 = 信息原语（3） + 控制原语（2）**

#### 信息原语（保持不变）
1. `ai_say`：信息输出（包含沉默这种特殊输出）
2. `ai_ask`：信息收集
3. `ai_think`：信息处理

#### 新增控制原语
1. `ai_wait_for`：条件等待
   ```yaml
   - action: ai_wait_for
     condition: "user_emotional_intensity < 7"
     timeout: 60
     on_timeout: next_action_id
   ```

2. `ai_branch`：条件分支
   ```yaml
   - action: ai_branch
     conditions:
       - when: "user_shows_avoidance"
         goto: gentle_probe
       - when: "user_shows_anger"
         goto: address_rupture
       - default: continue
   ```

**论据**：
- 编程语言的最小完备集 = 数据操作指令 + 控制流指令
- 咨询会谈的最小完备集 = 信息操作动作 + 流程控制动作
- 这仍然是5个基础动作，而非无限扩展

### 反方聚焦核心矛盾
即使加入流程控制，仍存在根本性的**抽象层次错配**问题：

#### 问题1：ai_say的语义范围过宽
对比以下场景中的"说"：
```yaml
场景A - 教育性解释：
  ai_say: "焦虑的神经生理机制是..."
  本质：知识传递

场景B - 共情性确认：
  ai_say: "听起来你感到被忽视了"
  本质：情感镜映

场景C - 治疗性沉默：
  ai_say: ""
  duration: 30
  本质：空间给予

场景D - 指导性干预：
  ai_say: "我需要你现在做三次深呼吸"
  directive: true
  本质：行为引导
```

这四个场景在咨询理论中属于**完全不同的干预类别**：
- A属于心理教育（Psychoeducation）
- B属于反映技术（Reflection）
- C属于治疗性使用自我（Therapeutic Use of Self）
- D属于危机干预（Crisis Intervention）

**将它们统一为ai_say + 参数，类似于将"手术"、"按摩"、"握手"统一为"触碰"动作**。

#### 问题2：ai_ask的方向性假设
ai_ask隐含"咨询师控制对话方向"的假设，但现代咨询理论强调：

- **来访者中心疗法**：来访者是专家，咨询师跟随
- **叙事疗法**：来访者是故事的作者，咨询师是见证者
- **对话式疗法**：咨询师-来访者是共同探索的平等关系

这些取向需要的不是"ask"（提问），而是：
- `ai_follow`：跟随来访者的叙事线索
- `ai_witness`：见证而不引导
- `ai_co_explore`：共同探索而非单向询问

#### 问题3：ai_think的可见性悖论
ai_think作为"内部处理"，在脚本中是否应该：
- **对来访者不可见**：纯粹的咨询师内心活动
- **对来访者可见**：透明化的思考过程（如"让我理解一下..."）

如果不可见，如何在脚本中触发咨询师的自我表露（"我注意到我自己有些焦虑..."）？
如果可见，它与ai_say的边界在哪里？

## 第四轮：实用主义与理论完备性的权衡

### 正方务实论证
从**工程实现和使用者认知负担**角度：

1. **认知简洁性**
   - 三个动作（+2个控制）的心智模型远优于12-15个领域特定动作
   - 脚本编写者只需掌握5个概念，而非记忆"共情"、"解释"、"见证"等的精确区分

2. **组合灵活性**
   ```yaml
   # 用基础动作组合实现"苏格拉底式共情探询"
   - action: ai_say
     content: "听起来你对这个决定有矛盾的感受"
   - action: ai_ask
     content: "矛盾的两面分别是什么？"
   - action: ai_think
     focus: "识别认知冲突的深层价值观"
   ```
   比预定义"ai_socratic_empathy"更灵活

3. **版本演进友好**
   - 通过扩展参数（向后兼容）而非新增动作（破坏性变更）
   - 避免"行为爆炸"（15个动作×10个参数 = 150种配置理解）

4. **实际充分性验证**
   - 挑战：找出一个真实咨询场景，无法用5个动作的组合序列实现
   - 迄今为止，反方未能提供这样的反例

### 反方理论坚守
从**领域建模准确性和可维护性**角度：

1. **领域语言的表达力**
   ```yaml
   # 方案A：基础动作组合（当前方案）
   - ai_say: "听起来你感到..."
     tone: empathic
   - ai_ask: "能多说说吗？"
     style: open_gentle

   # 方案B：领域动作（建议方案）
   - ai_reflect_feeling:
     observed_emotion: abandoned
     intensity: moderate
   - ai_invite_elaboration:
     focus: emotional_experience
   ```

   方案B的优势：
   - **语义自文档化**：动作名称即咨询理论术语
   - **约束正确性**：`ai_reflect_feeling`必须包含情感词汇，编译期可检查
   - **可分析性**：统计脚本使用了多少"反映技术"vs"封闭式提问"

2. **认知分层问题**
   - 基础动作方案强迫用户在**实现层**思考（"这里用say还是ask"）
   - 领域动作方案允许用户在**意图层**思考（"这里需要共情还是探询"）
   - 类比：SQL（意图）vs 手写索引遍历（实现）

3. **AI生成脚本的困境**
   ```
   提示词：生成一个认知行为疗法的苏格拉底式提问序列
   
   基础动作方案：AI需要"翻译"CBT理论 → 基础动作组合
   领域动作方案：AI可以"直接映射"CBT理论 → ai_socratic_question
   ```
   领域动作降低了从咨询理论到可执行脚本的转换成本

4. **边界案例的显式化**
   - 危机干预中的"ai_safety_contract"（安全契约建立）
   - 精神分析中的"ai_interpret_transference"（移情解释）
   - 这些是咨询师培训中的**独立技能单元**，不应降级为参数

## 第五轮：混合方案的可行性探讨

### 正方妥协提案
采用**分层动作系统**：

#### 核心层（Core Actions）
```yaml
ai_say      # 信息输出
ai_ask      # 信息收集  
ai_think    # 信息处理
ai_wait_for # 条件等待
ai_branch   # 条件分支
```
保持最小完备性，支持任意咨询场景

#### 扩展层（Extended Actions）
```yaml
ai_reflect_feeling    → 编译为: ai_say + tone:empathic
ai_socratic_question  → 编译为: ai_ask + style:socratic + ai_think
ai_psychoeducation    → 编译为: ai_say + tone:educational
ai_invite_elaboration → 编译为: ai_ask + style:open
```
提供领域语义糖，自动降级为核心动作

#### 优势
1. **用户可选择抽象层次**：新手用扩展动作，专家用核心动作
2. **引擎简化**：只需实现5个核心动作
3. **理论对齐**：扩展动作名称与咨询术语一致
4. **可扩展性**：用户自定义扩展动作（宏定义）

### 反方部分接受
分层方案可行，但需明确三个设计约束：

1. **扩展动作不应是纯语法糖**
   ```yaml
   # 错误示例：纯参数映射
   ai_reflect_feeling:
     emotion: sad
   # 等价于：
   ai_say:
     content: "你感到悲伤"
     tone: empathic
   
   # 正确示例：包含领域逻辑
   ai_reflect_feeling:
     observed_cues: ["低头", "声音颤抖", "用词消极"]
     inferred_emotion: sad
     confidence: 0.8
   # 编译时校验情绪词汇表，运行时记录推理依据
   ```

2. **核心动作应足够原子化**
   - 如果ai_say既要表达"说话"又要表达"沉默"，说明它不够原子
   - 建议分离：`ai_speak`（主动输出）vs `ai_hold_space`（治疗性沉默）

3. **扩展动作的标准化责任**
   - 需要"动作语义规范"，定义每个扩展动作的前置条件、效果、参数约束
   - 避免用户自定义的扩展动作破坏咨询理论的一致性

## 主持人总结：趋向共识的设计原则

### 辩论揭示的核心张力
1. **工程简洁性 vs 领域表达力**
2. **抽象通用性 vs 场景特异性**
3. **图灵完备性 vs 认知友好性**

### 提议的设计共识

#### 共识1：最小核心 + 可扩展层
**核心动作层（不可减少）：**
```yaml
ai_speak        # 主动语言输出
ai_inquire      # 主动信息收集
ai_process      # 内部信息处理
ai_hold_space   # 被动空间给予（治疗性沉默/等待）
ai_branch       # 条件分支
```
**5个原子动作**，放弃原ai_say的模糊语义

**扩展动作层（可插拔）：**
```yaml
ai_reflect_*     # 反映技术家族
ai_question_*    # 提问技术家族
ai_intervene_*   # 干预技术家族
ai_psychoeducate # 教育技术
...
```
编译为核心动作序列，但保留领域语义

#### 共识2：参数化的边界
- **核心动作**：参数控制执行细节（语速、停顿、必填项）
- **扩展动作**：参数体现咨询理论概念（情绪类型、探询深度）

**反模式**：用100个参数让ai_say模拟所有行为
**正模式**：用20个扩展动作+适度参数表达咨询技术

#### 共识3：验证可行性的三个测试
1. **完备性测试**：选取10个不同流派的咨询案例，验证是否可用5个核心动作表达
2. **可读性测试**：让咨询师（非程序员）阅读脚本，能否理解对话流程
3. **可生成性测试**：让LLM根据咨询理论描述生成脚本，评估理论-代码映射难度

#### 共识4：保留争议的设计空间
**存档待决问题：**
1. ai_think是否应该支持"可表达的思考"（think-aloud）模式？
2. 是否需要"ai_meta"动作表达关于咨询关系本身的操作？
3. 扩展动作是否应该允许用户运行时自定义，还是仅由引擎提供？

### 正方逻辑强度

### 反方逻辑强度

✓ 具体反例充分：沉默、移情、危机干预

✗ 过度坚持理论完美，忽略了工程权衡的必要性  

### 主持人裁决的逻辑合理性
1. **承认双方的有效论点**：不做非此即彼的选择
2. **寻找高阶解决方案**：分层架构化解矛盾
3. **保留设计空间**：承认当前信息不足以做最终决策
4. **提出可验证的测试**：用实证而非纯推理解决争议

**实现**：形成扩展动作规范，支持第三方贡献

**指标**：扩展动作的复用率、理论覆盖度

## 第六轮：基于LLM提示边界的三动作重构讨论

### 本轮分析主题
- **焦点1**：在仅保留ai_say、ai_ask、ai_think三类动作的前提下，利用LLM提示词中的主题与表达范围来刻画动作边界
- **焦点2**：为ai_say与ai_ask设计可验证的退出机制，使其在单次调用内自然收束而非无限追问/输出
- **焦点3**：明确ai_think作为“只写变量、不对用户说话”的内部认知动作的边界与作用范围
- **焦点4**：评估沉默与共情是否可以完全通过提示词约束在三类动作内部表达，而不新增动作类型

### 正方观点：三动作 + 提示边界足以收敛会谈

#### 1. LLM提示边界可以收缩动作语义
正方在前几轮接受“核心动作语义过宽”的批评，但认为在**LLM时代**可以通过精细提示边界对三类动作做“软收缩”：

- **ai_say 边界（陈述型输出）**：
  - 只允许产生**单轮陈述**，不包含问句、不引入新任务；
  - 提示中限定：\(\text{目的} = \) 解释 / 共情 / 总结中的一种或有限组合；
  - 明确“禁止内容”：不做决策、不收集新信息、不改变话题主轴；
  - 通过提示保证输出形态接近“咨询师的一小段话”。

- **ai_ask 边界（信息收集型输出）**：
  - 只输出问题或问题组，不进行长篇解释或建议；
  - 明确“本轮要补齐的变量/信息缺口列表”，要求LLM**只围绕缺口提问**；
  - 禁止追加“顺便安慰/教育”的语句，避免ask退化为混合动作；
  - 区分：开放式问句 vs 封闭式问句，可在提示中以“问题类型”参数化。

- **ai_think 边界（内部认知型动作）**：
  - 明确“这一步不直接对来访者说任何话”；
  - 提示约束输出为**结构化变量**（如`summary`, `hypothesis`, `risk_level`），而非自然语言话术；
  - 只读取历史片段与变量，不创建新的对话回合；
  - 作为后续ai_say/ai_ask的“认知底稿”，但本身不面向用户。

从这个角度看，**问题不再是动作是否过宽，而是提示是否足够精确**。

#### 2. 沉默与共情可以内嵌在提示语义中
正方认同“沉默/共情是关键咨询技能”，但认为仍可内嵌到三种动作之中：

- **沉默**：
  - 作为`ai_say`的一种极端形态：内容为空、但在提示中声明“此时的沉默具有治疗性目的”；
  - 引擎层通过`min_silence_duration`、`max_silence_duration`控制时间边界；
  - LLM只负责生成“沉默意图描述/标签”，由系统在UI/时间轴上渲染。

- **共情**：
  - 视为`ai_say`的一种**话术风格**（tone: empathic），在提示中固定其优先级；
  - LLM根据`target_emotion`（来自ai_think或变量）生成对应的镜映语句；
  - 不额外引入“ai_empathize”动作，而是在ai_say内部用参数锁定其语义。

在这种设计下，**三动作模型仍然保持不变，沉默与共情只是ai_say的受限子模式**，由提示边界保证不越界。

#### 3. 退出机制可以通过“有限目标 + 单步约束”实现
正方认为，退出机制可以在**动作配置 + LLM提示**的组合下得到收敛：

- **ai_say退出机制**：
  - 每次调用只允许生成**一段话**，不展开多轮对话；
  - 提示中显式要求“输出结束后停止，不再提出问题或引出新任务”；
  - 引擎层可以加上`max_tokens`约束，避免冗长解释。

- **ai_ask退出机制**：
  - 每次调用绑定一个**有限的信息缺口集合**，如`[当前情绪强度, 触发诱因]`；
  - LLM被要求在有限数量（如1-3个）问题内尽量补齐这些缺口；
  - 输出结构中增加`need_followup: true/false`，交由Topic层决策是否再来一轮ai_ask，而非让LLM在单步中自循环。

- **ai_think退出机制**：
  - 仅允许读取指定历史窗口，不允许“回溯过多轮”；
  - 输出变量结构固定，如：
    - `summary`（简要总结）
    - `key_themes`（主题列表）
    - `hypotheses`（若干假设+置信度）
  - 以“填满结构化槽位”为退出条件，而不是以自然语言长度为标准。

在这些约束下，Topic层或更高层的智能可以把“生成动作序列”的任务交给LLM，但每个动作仍然是**一次性、可收束、易验证**的调用。

---

### 反方观点：仅靠提示边界存在模糊与不可验证风险

#### 1. LLM遵守边界的可验证性不足
反方强调，**边界如果只写在自然语言提示里，就很难做程序级验证**：

- LLM可能在`ai_ask`中夹带安慰性陈述，使其部分变成ai_say；
- 即使提示写明“不要解释”，LLM在不确定时仍可能生成短解释以自我合理化；
- 对于“不要引入新任务/话题”的约束，人类也难以给出清晰的判定标准，更不用说自动检验。

因此，如果三种动作的边界**完全依赖提示中的语义约定**，则：
- Topic层在调度动作时很难保证“这个动作到底执行了什么”；
- 事后分析时，无法可靠地统计“本轮用了多少次提问/共情”等干预手段；
- 安全场景（如危机风险）下，单靠提示难以保证必然触发某些动作或必然停止提问。

#### 2. 沉默与共情的提示表达容易漂移
沉默与共情如果仅作为ai_say内部的一种“风格”，反方认为会有两个问题：

1. **语义模糊**：
   - “在适当时刻保持沉默并共情”是一种高度模糊的描述；
   - 不同LLM版本、不同提示细节，会导致行为显著差异；
   - 对话重现性差，不利于临床监督或质量评估。

2. **模型更新敏感**：
   - 一次LLM升级可能改变其对“共情语气”的理解；
   - 若没有更硬的行为接口（如扩展动作`ai_reflect_feeling`），现有脚本语义可能“被动漂移”。

反方认为，**真正重要的临床技能（沉默、共情、危机干预）不应该完全交给提示的软约束，而应在动作层或至少在扩展动作层有更硬的刻画**。

#### 3. 退出机制如果只写在提示中，难以形成Topic层的清晰协议
Topic层希望把“生成动作序列”的任务交给LLM，本质上需要一个**清晰的协议**：

- Topic层："请给我一串ai_ask、ai_say、ai_think动作"
- LLM：需要知道每种动作**具体允许做什么、不允许做什么，以及何时停止**。

如果这些规则只是大量自然语言提示：
- 协议本身不具备机器可读的结构；
- 不同Topic之间难以复用同一套“动作边界”；
- 日后演进（如增强ai_ask的能力）需要改动多个提示，风险难控。

反方因此主张：
- 即使动作仍然只有三种，也应该在引擎中为每个动作定义一份**机器可读的边界描述（schema + 约束）**；
- 提示只是对这份边界的“自然语言解释”，而不是边界本身；
- 退出机制尤其需要明文化成状态机，而不是单一提示句。

---

### 主持人小结：面向Topic层的“动作边界层”设计

#### 1. 三动作共识与新增视角的整合
结合前五轮与本轮讨论，可以形成一个折衷观点：

- 在**引擎执行层**，继续坚持三类原子动作：`ai_say`、`ai_ask`、`ai_think`；
- 在**边界定义层**，为每个动作增加一份结构化的“边界与退出协议”；
- 在**提示实现层**，让LLM按照“边界协议”生成具体话术或内部变量；
- 沉默与共情可以优先尝试通过ai_say的子模式实现，但对其行为边界要有结构化定义，以便后续视情况上升为扩展动作。

#### 2. 面向三类动作的边界草案（供后续迭代）

- **ai_say：陈述型动作**
  - **允许**：
    - 围绕指定主题进行解释、反馈、共情、总结；
    - 通过子模式表达沉默（空内容 + 时间边界）、简短情感镜映；
  - **禁止**：
    - 主动提出新问题；
    - 引入新的会谈任务或作业；
    - 试图在同一调用中“又解释又收集信息”；
  - **退出条件（示例）**：
    - 完成一次性输出（单段或有限几段）；
    - 不超过配置的token/句子上限；
    - 输出结构满足“无问句、无新任务提示”的校验规则。

- **ai_ask：收集型动作**
  - **允许**：
    - 围绕显式列出的信息缺口提出问题；
    - 以开放/封闭/量表等形式收集数据；
  - **禁止**：
    - 给出长篇解释或建议；
    - 评判、贴标签或改变话题主轴；
  - **退出条件（示例）**：
    - 信息缺口列表全部被填充或达到提问数上限；
    - 输出结构中显式返回`need_followup = false`；
    - 引擎层统计“连续ai_ask”次数，超过阈值则强制切换到ai_think或ai_say。

- **ai_think：内部认知动作**
  - **允许**：
    - 对限定窗口内的历史对话做总结、主题提取、假设构建；
    - 写入内部变量供后续动作引用；
  - **禁止**：
    - 直接对用户说话；
    - 更改对话元数据（如会谈边界）而不通过显式动作；
  - **退出条件（示例）**：
    - 填满预定义变量槽位（summary、themes、hypotheses等）；
    - 不超过配置的计算窗口和token预算。

这些边界本身应以**schema+约束**形式固化在系统中，再由提示将其翻译为LLM可以理解的自然语言。

#### 3. 面向后续“Topic层/更高层智能”的启发

基于本轮讨论，可以给出一个面向Topic层的初步设计方向：

1. Topic层不直接与“提示文本”打交道，而是与一组**动作描述+边界约束**打交道；
2. Topic层只需决定“在当前状态下，调用哪种动作、带哪些参数”，而不关心具体话术；
3. LLM负责在每个动作的边界内生成话术或内部变量；
4. 日后若需要引入扩展动作（如`ai_reflect_feeling`），可以定义为：
   - 一个**固定的动作组合模板**（若仍坚持三动作模型），或
   - 一个在边界层有独立schema的“二层动作”。

### 场景1：建立关系与初步探索

#### 对话原文
```
咨询师：林薇，你好。很高兴再次见到你。上一周过得怎么样？有没有什么特别想和我分享的？
来访者：(苦笑一下) 说实话，不太好。感觉……又回到原点了。
咨询师：嗯，这种感觉在咨询过程中是常见的，就像爬山，有时会遇到一些坡坎，感觉退回去了，但这不意味着之前的努力白费了。愿意和我具体说说，是什么事情让你有"回到原点"的感觉吗？

#### 动作拆解

  - 信息缺口：本周发生了什么、来访者当前状态

  question_count: 1-2

  - 退出条件：提出1个具体化问题后立即结束

  **信息缺口**：具体是什么事件导致了这种感觉

expected_output:
  question_count: 1
  question_style: "specific, event-focused"
  need_followup: true
```

### 验证结论

#### 1. 充分性验证
通过对完整CBT会谈的拆解，三动作模型能够覆盖：
- **信息收集**（ai_ask）：开放探询、具体化、追踪思维链、证据检验、童年探索
- **信息输出**（ai_say）：共情、正常化、心理教育、框架设定、策略传授、会谈总结
- **内部处理**（ai_think）：模式识别、策略规划、风险评估、会谈总结

**未发现无法表达的咨询行为**。沉默、共情等均可通过ai_say的子模式 + 提示约束实现。

#### 2. 边界与退出机制的有效性
- **ai_ask** 的退出机制：每次绑定明确的信息缺口（1-3个），提问后立即结束
- **ai_say** 的退出机制：限定句子数量（2-6句），禁止提出新问题
- **ai_think** 的退出机制：输出固定结构的变量槽位，不产生面向用户的文本

通过提示词的明确约束（"允许"、"禁止"、"退出条件"），可以有效控制单次动作的范围。

#### 3. 提示词设计的关键要素
每个动作的提示词包含：
1. **purpose**（本动作的咨询目标）
2. **边界声明**（允许/禁止/退出条件）
3. **上下文变量**（已知信息、信息缺口）
4. **输出约束**（句子数、语气、是否包含问题）
5. **CBT技术标签**（如有，如"socratic_questioning"）

#### 4. Topic层调度的可行性
Topic层可以基于：
- 当前会谈阶段（建立关系 → 探索问题 → 干预 → 总结）
- 信息完整度（CBT模型的五要素是否收集齐全）
- 干预深度（表层应对 → 自动思维 → 核心信念）

来决定下一个动作的类型、目标、信息缺口，将这些作为参数传递给LLM生成具体话术。
