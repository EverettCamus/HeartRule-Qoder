# Story 2.2：Topic层基于实体列表动态展开 Action 队列的智能能力需求

## 1. 能力背景

在 HeartRule 五层架构中，Topic 层承担“目标攻坚单元”的角色：围绕一个清晰子目标（例如“收集童年主要抚养者信息”），编排和调整一组 Action 队列。

在传统的“静态脚本”模式下，Topic 下的 Action 列表是预先写死的：

- **优势**：流程可控、易于调试。
- **局限**：无法根据现场新出现的对象（如更多抚养者、更多部门/角色）动态调整 Action 队列，只能用人工写很多分支或重复片段，既不优雅也不智能。

为了承载更接近真实咨询现场的复杂展开需求，本 Story 希望在 Topic 层引入一个 **可调用 LLM 的“意识层”**：

- 在 Action 执行过程中，持续关注对话与变量的变化；
- 一旦发现“可枚举实体列表”被识别/更新（如多位抚养者、多部门、多角色），就能 **按同一套路为每个实体实例化一组 Action 子队列，并动态插入到当前 Topic 的执行计划中**。
- 上述“按同一套路展开”的动作本身也应由 Topic 层的 strategy/意识层按需触发：当判断当前目标不适合展开、或需要优先处理其他任务时，可以选择不展开或只对部分实体展开，而不是固定写死为“发现列表就必然全部展开”。

## 2. 能力目标

**本 Story 的能力目标可以概括为：**

1. **感知实体列表**：Topic 意识层能够从对话或变量中识别出“需要逐一处理的一组实体”（如父亲、母亲、外公）。
2. **复用一套脚本节奏**：对每个实体重复同一套 Action 模板（如“基本信息 → 关系史 → 关键事件”），而非人工复制多份脚本。
3. **动态扩展队列**：在 Topic 已经开始执行的过程中，如果后续 Action 新识别出更多实体（如在“父亲”小节里又发现“外公”），可以 **追加新的实体子队列**。
4. **保持执行秩序清晰**：同一实体的 Action 子队列在执行上保持相对连续，不让“父亲 / 母亲 / 外公”的提问交叉打散，避免用户体验上的混乱。
5. **可观测、可调试**：所有展开与调整决策（识别了哪些实体、是否展开、如何展开、插入到哪里）都有清晰的 debug 信息记录，方便在脚本调试工具中复盘。
6. **策略驱动而非硬编码**：是否需要动态调整 Action（包括是否展开实体子队列、展开到什么程度），由 Topic 的 strategy/意识层在每次规划时综合当前目标、信息充分度、时间等因素做出判断，引擎只提供通用的队列操作能力而不写死具体业务规则。

> 注：本 Story 聚焦在 **“一对多实体 + 队列展开”** 这一能力，不处理阻抗检测、时间预算、优先级重排等问题，这些在后续 Story 中承担。

## 3. 典型使用场景

### 3.1 CBT 咨询：多抚养者信息收集

**Topic 目标**：收集来访者童年主要抚养者情况，包括：

- 有哪些抚养者？
- 各抚养者的基本信息（年龄、称呼、亲属关系、职业等）。
- 各抚养者与来访者之间的关系史和关键事件。

**脚本初始设计（抽象模板）：**

- **Action1**：ai_ask — 询问有哪些抚养者，收集“抚养者列表”。
- **Action2 模板**：ai_ask — 询问某一位抚养者的基本信息。
- **Action3 模板**：ai_ask — 询问该抚养者与来访者的关系史与关键事件。

**动态展开期望：**

1. 在 **Action1** 中，来访者第一次提到“爸爸”。
   - Topic 意识层识别到 `抚养者列表 = [爸爸]`。
   - 在当前 Topic 队列中，按模板插入：
     - Action2(爸爸)
     - Action3(爸爸)

2. 在执行 **Action3(爸爸)** 的过程中，来访者又提到“妈妈、外公”。
   - Topic 意识层检测到 `抚养者列表` 发生增量：从 `[爸爸]` 变为 `[爸爸, 妈妈, 外公]`。
   - 为新增实体生成新的子队列，并按照约定顺序 **追加到当前 Topic 队列后面**：
     - Action2(妈妈)
     - Action3(妈妈)
     - Action2(外公)
     - Action3(外公)

3. 最终，Topic 内的 Action 执行顺序类似：
   - Action1
   - Action2(爸爸)
   - Action3(爸爸)
   - Action2(妈妈)
   - Action3(妈妈)
   - Action2(外公)
   - Action3(外公)

**关键点：**

- 脚本作者只需要写一次 Action2 / Action3 模板。
- 实体列表的识别和队列展开全部由 Topic 意识层负责。
- 在会谈过程中发现新抚养者时，系统自动追加对应脚本片段，而无需预先穷举所有亲属组合。

### 3.2 多人物 / 多部门信息收集

相同的能力可以推广到其他典型场景：

- 了解来访者的团队成员：对每个人重复“角色定位 → 配合模式 → 冲突点”三个 Action。
- 职场咨询中，对多个关键干系人（领导、同事、下属）分别收集关系史与事件。
- 企业场景中，对多个部门分别收集现状、痛点、期望等信息。

这些场景的共同特点：

- 存在清晰的“实体集合”（人、组织、情境等）；
- 对每个实体，需要按照同一节奏进行一轮或数轮固定结构的对话；
- 实体集合可能在对话过程中 **被追加或修正**。

## 4. 核心概念与角色

- **实体（Entity）**：需要被逐一处理的对象，如“爸爸、妈妈、外公”、“销售部、研发部”等。
- **实体列表（Entity List）**：当前 Topic 下，与某一对子任务模板相关联的一组实体。
- **Action 模板（Action Template）**：针对单个实体的一小段 Action 子序列，例如“问基本信息 → 问关系史 → 问关键事件”。
- **Topic 意识层（Topic Cognitive Layer）**：
  - 可调用 LLM；
  - 负责理解对话上下文、变量快照；
  - 决定何时、以何种方式基于实体列表展开 Action 队列。

## 5. 功能需求拆解

### 5.1 实体感知与列表管理

1. **来源**：实体可以来自一个或多个渠道：
   - 当前 Action 的 output 变量（如 `抚养者列表`）。
   - 最近几轮对话中由 LLM 直接解析出的称呼/角色。
   - 之前 Topic 或 Session 存下的结构化变量。

2. **增量更新**：
   - Topic 意识层需要支持“新增实体”的概念，而不仅仅是一次性识别：
     - 每次 Action 执行后，对相关变量进行比对，识别出新出现的实体；
     - 避免重复处理已完成的实体。

3. **去重与规范化**：
   - 将“爸、父亲、爸爸”规范为同一实体；
   - 对相同称呼但不同指代的情况（如“两个姨妈”）可由后续 Story 细化，本 Story 只要求基本去重能力。

### 5.2 Action 模板声明与绑定

1. **脚本层声明**：
   - Topic 脚本需要有一种方式声明：
     - 哪一段 Action 列表是“针对单个实体的模板片段”；
     - 模板里有哪些位置会引用“当前实体”的属性（如 `{抚养者称呼}`、`{与该抚养者的关系}`）。

2. **单个实体的语义绑定**：
   - 对于模板内的每一个 Action：
     - 提示词中可以使用“当前实体”的相关变量进行占位；
     - output 中的变量名可以根据实体名称进行区分（如 `爸爸_关系史`、`妈妈_关系史`）。

3. **模板与 Topic 策略的关系**：
   - Topic 可配置执行策略（例如“至少完成一个核心抚养者；有时间再展开其他人”），本 Story 只要求对每个实体都有对应的模板队列，是否全部执行交由上层策略 Story 控制。

### 5.3 Action 队列展开与插入策略

1. **初次展开**：
   - 当实体列表第一次建立（如 Action1 收集到第一个或多个抚养者）时：
     - Topic 意识层根据实体列表，将模板 Action 复制为多组实体子队列；
     - 默认按“实体顺序 × 模板顺序”展开，如：
       - 实体1：A2(实体1), A3(实体1)
       - 实体2：A2(实体2), A3(实体2)
       - …

2. **过程中追加**：
   - 当 Topic 已经在执行某个实体的小节时（例如正在执行 A3(爸爸)），如果识别到新实体：
     - 将新增实体的子队列 **追加到当前 Topic 队列的末尾**；
     - 保留已规划队列的相对顺序，不打乱之前的执行计划。

3. **执行连续性要求**：
   - 同一实体的 Action 子队列应尽量连续执行：
     - 避免出现 “A2(爸爸) → A2(妈妈) → A3(爸爸) → A3(妈妈)” 这种交叉模式；
     - 默认策略为：一个实体的一组 Action 执行完，再进入下一个实体。

4. **与其他 Topic 调整能力的关系**：
   - 本 Story **只处理“在当前 Topic 内追加实体子队列”**；
   - 不涉及跨 Topic 跳转、插入临时 Topic 或重排 Topic 顺序。

### 5.4 与 Topic 意识层（LLM）的协作方式

1. **LLM 参与点**：
   - 识别对话中的实体候选（如“爸爸、妈妈、外公”）；
   - 在变量更新时，判断是否应该视为新的实体；
   - 对模板展开过程本身的控制逻辑（循环、去重）应尽量保持在引擎代码中，而非交由 LLM 决策。

2. **约束**：
   - Topic 意识层在本能力中 **不修改单个 Action 的业务逻辑**（提示词内容仍由脚本/Action 层负责）；
   - 它的职责是：
     - 识别“要对哪些实体重复这些 Action”；
     - 决定“在队列中何处追加对应实体的子队列”。

### 5.5 Topic 策略层三分类决策与双提示词模板

在 Story 2.2 中，引擎层需要为 Topic 的 strategy/意识层提供一个统一的“决策+生成”机制，而不是在代码里写死具体策略。整体采用 **三分类决策 + 双提示词模板** 的方式：

1. **三类决策（由策略层 + 提示词模板 A 做出）**：
   - `NO_CHANGE`（不调整）：继续按现有 Action 队列执行，不改变队列结构，也不强制重写当前 Action，只在微小范围内允许运行时的表达差异。
   - `TUNE_CURRENT_ACTION`（仅调整当前 Action 策略）：认为 Topic 整体脚本方向正确，但当前这一步的提问/表达方式与来访者状态略不匹配，只需要在语气、问法、深浅或聚焦点上做轻微调整，而**不改变后续 Action 队列结构**。
   - `REPLAN_ACTION_QUEUE`（需要调整 Action 队列）：认为仅微调当前 Action 已不足以满足 Topic 策略，需要在当前 Topic 内对后续 Action 队列进行结构性调整，例如：
     - 针对一组实体（抚养者、部门、角色等）生成/插入成组的 Action；
     - 插入新的小节、跳过部分 Action，或重排执行顺序。

2. **提示词模板 A：决策层模板（Decision Prompt Template）**
   - 目标：在上述三类决策中选择其一，并给出简要理由和后续指引。
   - 由引擎灌入的关键信息包括：
     - 当前 Topic 的 `goal` 与 `strategy` 描述；
     - 当前正在执行的 Action 概要（类型、核心提示词、输出目标、轮次信息等）；
     - 当前 TopicPlan 概要（已执行 / 未执行 Action 列表）；
     - 最近对话摘要、已识别实体及其处理进度、剩余时间与来访者状态等。
   - 输出（结构化 JSON，便于引擎消费）：
     - `decision`: "NO_CHANGE" \| "TUNE_CURRENT_ACTION" \| "REPLAN_ACTION_QUEUE"；
     - `reason`: 简要中文原因；
     - `tune_current_action_hint`: 当决策为 TUNE_CURRENT_ACTION 时，对“当前 Action 应如何微调”的方向性提示（不直接生成脚本）；
     - `queue_replan_hint`: 当决策为 REPLAN_ACTION_QUEUE 时，对“要在哪些层面调整队列”的意图描述，例如是否涉及多实体展开、目标实体列表以及约束条件（时间、优先级等）。

3. **提示词模板 B：队列生成模板（Queue Replan / Expansion Prompt Template）**
   - 仅在模板 A 判定为 `REPLAN_ACTION_QUEUE` 时触发。
   - 目标：在已经确定“需要调整 Action 队列”的前提下，根据 Topic 策略与约束，生成一段**符合脚本 schema 的 Action 片段**（例如针对多抚养者的一组 `ai_ask` 序列），供引擎进行 schema 校验后插入。
   - 由引擎灌入的关键信息包括：
     - Topic 的 `goal` / `strategy`；
     - 来自模板 A 的 `queue_replan_hint`（调整意图、是否为多实体展开、目标实体列表等）；
     - 针对“单个实体应如何提问”的节奏说明（例如：先基本信息→再关系情况→再一段印象深刻事件）；
     - 动作与 schema 约束（允许的 action 类型、必填字段、每个实体最多多少个 action 等）；
     - 新生成 Action 将被插入到当前 Topic 队列中的位置描述（例如“在现有队列末尾追加”）。
   - 输出：
     - 一段仅包含 `actions` 列表的 YAML/JSON 片段（如 `actions: [...]`），每个 Action 都满足：
       - id 在本片段内唯一且具备可读语义；
       - type 只在允许的集合内（如 `ai_ask`、`ai_say`、`ai_think`），并遵守各自的语义边界；
       - config 与 output 字段符合既有 Topic 脚本 schema 约束。

4. **引擎层的职责边界**：
   - 引擎层负责：
     - 在适当的时机（例如每个 Action 执行前，或 Topic 内特定检查点）调用模板 A，获取三类决策之一；
     - 当 A 判定为 `REPLAN_ACTION_QUEUE` 时，调用模板 B 获取新增的 Action 片段；
     - 对模板 B 的输出进行 schema 校验，并以通用原语（insert/append/replace 等）将其合并进当前 Topic 队列。
   - 引擎层**不负责**：
     - 决定具体哪一类决策更符合咨询策略（这一点由 Topic strategy + 模板 A 来完成）；
     - 在代码中写死“发现实体列表就必然展开”等业务规则。

   #### 模板 A：Topic 决策层提示词模板（Decision Prompt Template）

   ```text
   你是一个负责【话题层规划】的咨询引擎助手，任务是在每一步执行前，判断是否需要调整后续的 Action 计划。

   【当前话题信息】
   - 话题目标：{%topic_goal%}
   - 话题策略：{%topic_strategy%}

   【当前执行状态】
   - 当前 Action 概要：{%current_action_summary%}
   - 当前话题计划概要（已执行/未执行）：{%current_topic_plan_summary%}
   - 已识别实体及处理进度（如有）：{%entity_list_status%}
   - 会谈剩余时间与来访者状态：{%time_and_state%}

   【对话上下文摘要】
   - 最近若干轮对话的关键信息：{%conversation_summary%}

   【三种可选决策及边界】
   1. NO_CHANGE（不调整）
      - 含义：继续按照现有 Action 队列执行，不改队列、不重写当前 Action。
      - 使用场景：当前问答方式和后续计划都与话题目标、策略匹配，没有新的结构性信息或明显阻碍。

   2. TUNE_CURRENT_ACTION（只调整当前 Action 策略）
      - 含义：不改后续队列结构，只微调“当前这一步该怎么问/怎么说”（如语气、问法、深浅、聚焦点）。
      - 使用场景：整体脚本方向正确，但当前提问方式与来访者状态不匹配，可能太难、太快或太绕。

   3. REPLAN_ACTION_QUEUE（需要调整 Action 队列）
      - 含义：需要对后续 Action 队列做结构性调整，例如：
        - 新出现了一组需要逐个处理的实体（抚养者、部门、角色等），需要为它们生成/插入成组的 Action；
        - 需要插入新的小节、跳过部分 Action 或重排顺序，以更好满足当前话题目标。
      - 使用场景：仅调当前 Action 已不足以满足策略，需要改变后面几步的行动方案。

   【任务】
   1. 仔细分析上述信息，在三种决策中选择**唯一一种**最合适的。
   2. 严格输出 JSON，字段如下：
      {
        "decision": "NO_CHANGE",
        "reason": "用不超过50字说明原因（decision 只能在 'NO_CHANGE'、'TUNE_CURRENT_ACTION'、'REPLAN_ACTION_QUEUE' 三者中选择其一）",
        "tune_current_action_hint": {
          "focus": "当 decision='TUNE_CURRENT_ACTION' 时，更需要调整语气/问法/深度/范围中的哪一类",
          "suggested_direction": "给出方向性建议，不写具体提示词"
        },
        "queue_replan_hint": {
          "need_entity_expansion": true,
          "target_entities": ["父亲", "母亲", "……"],
          "replan_intent": "概述希望如何调整队列，例如：为每个新增抚养者补一轮‘基本信息+关系史’",
          "constraints": "任何需要遵守的约束，如时间、优先级等"
        }
      }

   只输出 JSON，不要解释或添加多余文字。
   ```

   ````text
   你是 HeartRule 对话引擎中的【Topic规划助手】，任务是在已经确认需要“调整 Action 队列”的前提下，根据给定的策略与约束，生成一段新的 Action 脚本片段，用于插入到当前话题中。

   - 话题策略：{%topic_strategy%}
   ````

### 5.6 决策职责划分：Topic 意识层 vs Action 主 LLM

在 Story 2.2 中，“是否需要调整、以及是仅调整当前 Action 策略还是重排整个 Action 队列”的决策，统一放在 **Topic 层的策略/意识 LLM** 中完成，而不是分散在每个 Action 的主 LLM 提示词里。这一设计与《HeartRule 咨询智能实现机制》中“抬头规划，低头执行”的五层架构保持一致：

1. **Action 层（低头执行）只做什么**：
   - 根据预置脚本与少量参数，完成一次最小对话交互（提问、解释、总结等）；
   - 在边界内进行微观话术调整（例如语气、比喻、解释深浅），但不自行决定“是否重写后续脚本计划”。

2. **Topic 层（抬头规划）负责决定什么**：
   - 在关键节点或周期性检查点，由引擎调用提示词模板 A，对当前 Topic 状态做出三类决策之一：
     - `NO_CHANGE`：不调整队列；
     - `TUNE_CURRENT_ACTION`：仅调整当前 Action 的策略；
     - `REPLAN_ACTION_QUEUE`：需要对后续 Action 队列做结构性调整（例如针对多实体展开子队列）。
   - 当决策为 `REPLAN_ACTION_QUEUE` 时，再调用提示词模板 B，生成符合脚本 Schema 的 Action 片段，交由引擎进行 Schema 校验并插入队列。

3. **“调整当前 Action 策略”的分工方式**：
   - 是否需要调整当前 Action，由 Topic 层通过模板 A 决定，并在 `tune_current_action_hint` 中给出方向性提示（例如更需要调整语气/问法/深度/范围中的哪一类）；
   - Action 层的主 LLM 只需根据这些提示，在预设的若干表达模式中进行选择或轻量变体生成，而不自行改变队列结构。

4. **这样分层带来的好处**：
   - 保持 Action 层的提示词简单、稳定，便于在不同 Topic 中复用；
   - Topic 层可以更好地综合考虑话题目标、信息充分度、时间预算等因素，做出符合咨询策略的重规划决策；
   - 所有与“队列调整”相关的思考过程都集中在模板 A/B，易于观测、调试和跨 Story 复用（如 2.3 阻抗处理、2.4 信息充分度控制等后续能力）。

**3. 变量替换与使用机制（概览）**

1. 模板管理器在加载 `_system/config/...` 下的 default/custom 模板时，只保留 `{%var%}` 形式的系统变量占位符，不直接拼接运行时数据。
2. 每次实际调用模板 A/B 前，脚本执行引擎会基于当前 **Session / Phase / Topic / Action 执行上下文** 以及上一轮 LLM 决策结果，组装出一份 `SystemVars` 映射表：`{ topic_goal, topic_strategy, ... }`。
3. 模板管理器先替换系统变量（`{%var%}`），再替换脚本变量（如 `{抚养者称呼}` 等），生成最终发送给 LLM 的完整提示词文本。
4. 模板 A 的输出（尤其是 `queue_replan_hint` 部分）会被结构化存入执行上下文，供后续调用模板 B 时作为输入变量来源之一。
5. 模板 B 生成的 `actions` 片段在被引擎接收后，会再次经过 JSON Schema 校验与必要的安全检查，只有通过校验的部分才会被合并进 Topic 队列。

## 6. 可观测性与调试需求

1. **展开记录**：每次实体列表变化和队列展开，都需要在 debug 信息中记录：
   - 当前 Topic 名称；
   - 触发展开的实体列表（含新增部分）；
   - 使用的模板 Action 范围；
   - 展开前后的队列长度与示意顺序。

2. **实体状态标记**：
   - 对每个实体，记录其对应子队列的执行状态：未开始 / 进行中 / 已完成 / 被跳过；
   - 便于在脚本调试页面上直观看到“哪些抚养者已经聊完了，还有哪些尚未覆盖”。

3. **问题诊断辅助**：
   - 当出现“某个实体重复被展开”或“新增实体未被展开”时，调试信息能够帮助定位：
     - 是实体识别错误；
     - 还是展开逻辑/去重逻辑问题。

## 7. 范围外与后续演进

**本 Story 不解决但需要为后续预留空间的能力：**

1. **阻抗与情绪风险处理**（后续 Story 2.3）：
   - 在某个实体的小节里遇到强烈情绪或阻抗时，是否跳过后续 Action 或插入安抚 Topic。

2. **信息充分度控制**（后续 Story 2.4）：
   - 若时间有限，是否只对部分实体执行“深度版脚本”，其他实体执行“简略版脚本”。

3. **Topic 级时间预算与优先级重排**（后续 Story 2.5/2.6）：
   - 当收集的实体数量远超预期时，如何调整执行顺序或裁剪部分实体的脚本。

本 Story 的定位是在 Story 2.1 的基础上，为 Topic 层补上一块 **“一对多实体的队列展开能力”**，作为后续更复杂 Topic 智能（阻抗处理、策略性取舍）的基础积木。
