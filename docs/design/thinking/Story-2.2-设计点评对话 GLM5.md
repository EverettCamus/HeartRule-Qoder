# Story 2.2 设计点评对话 GLM5

**参与者**：

- 吴恩达（Andrew Ng）人工智能专家
- Martin Fowler 软件架构大师

**点评主题**：Topic 层 Action 队列动态调整的智能能力需求设计

---

## 第一轮：宏观印象

**吴恩达**：我先说说对这个设计的整体看法。Topic 层引入"意识层"这个概念很有意思——把 LLM 作为一个决策组件嵌入到流程编排中，而不是简单地用它来生成文本回复。这实际上是"Cognitive Architecture"的一种实践，让我想起了 ReAct 模式：感知-推理-行动的循环。

**Martin Fowler**：我从一个架构师的角度来看。这个设计有几个值得肯定的点：

1. **清晰的职责边界**：Action 层"低头执行"，Topic 层"抬头规划"，这句话说得很到位。单一职责原则在这里体现得很好。
2. **开闭原则**：通过"场景A作为首实现，框架支持扩展"的方式，为后续演进留出了空间。
3. **决策分离**：把"是否调整队列"的二分类决策和"如何调整"的细节分开，这是典型的策略模式思想。

**吴恩达**：同意。不过我要提一个问题——**二分类决策的选择**。设计中只有 `NO_CHANGE` 和 `REPLAN_ACTION_QUEUE` 两个选项。这个粒度是否足够？

---

## 第二轮：决策粒度讨论

**吴恩达**：从 AI 的角度，我关心决策的难度和可靠性。二分类决策的好处是简单、容错率高——即使模型偶尔判断错误，影响也相对可控。但问题在于：

- 如果一个场景需要"轻微调整"而非"完全重规划"，模型该怎么判断？
- `REPLAN_ACTION_QUEUE` 的语义是否太粗糙？

**Martin Fowler**：我注意到了设计中有一个关键的设计选择——去掉了 `TUNE_CURRENT_ACTION`。

> 注：去掉了 `TUNE_CURRENT_ACTION`，因为单个 Action 内部的微调由 Action 层通过多轮对话自身能力解决，Topic 层只负责判断是否需要调整队列结构

这个设计是合理的。如果 Topic 层和 Action 层都具备了"微调"能力，就会产生职责重叠和冲突。让 Action 层自己消化微调需求，保持了层的独立性。

**吴恩达**：我同意这个判断。这其实是"最小知识原则"在 AI 系统设计中的应用——Topic 层不应该知道 Action 内部的细节，只关心队列结构是否需要调整。

---

## 第三轮：提示词模板设计

**吴恩达**：让我看看提示词模板的设计。输出是 JSON 格式：

```json
{
  "decision": "NO_CHANGE" | "REPLAN_ACTION_QUEUE",
  "reason": "原因说明",
  "queue_replan_hint": {...}
}
```

这个设计有几个 AI 工程方面的考量：

**优点**：

1. 结构化输出便于程序解析和处理
2. `reason` 字段强制模型解释决策，有助于可调试性
3. `queue_replan_hint` 提供了调整意图的语义描述

**潜在风险**：

1. 当决策为 `NO_CHANGE` 时，`queue_replan_hint` 字段是多余的
2. 提示词模板需要灌入多个变量（Topic goal、对话摘要、对象列表状态等），信息量较大，可能影响推理质量

**Martin Fowler**：从可维护性角度，我建议将模板拆分。决策层模板只负责"做决策"，执行层模板负责"生成调整方案"。这样可以：

1. 降低单次推理的复杂度
2. 按需调用——不是每次都需要生成详细的调整方案
3. 便于独立测试和调优

**吴恩达**：这个建议很好。这其实是"Chain-of-Thought"和"工具调用"的结合——先让模型判断是否需要调整，如果需要，再调用一个专门的"规划工具"来生成调整方案。这样可以提高推理质量。

---

## 第四轮：对象感知与列表管理

**Martin Fowler**：我注意到 5.1 节的"对象感知与列表管理"需求。这一块涉及：

1. 来源多样化：变量、对话解析、其他 Topic
2. 增量更新和去重

从架构角度，我关心这些能力应该放在哪一层？

- 如果放在 Topic 意识层，会增加意识层的复杂度
- 如果单独抽一个"对象管理器"组件，需要在 Topic 和这个组件之间建立清晰的接口

**吴恩达**：从 AI 实践角度，"去重"是一个有挑战的问题——"爸、父亲、爸爸"需要被识别为同一对象。这不仅仅是字符串匹配，需要语义理解。

我的建议是：

1. **对象提取**：专门用一个 LLM 调用来做 NER（命名实体识别）+ 语义归一化
2. **对象状态管理**：用传统的数据结构来维护，不要让 LLM 承担这类确定性的逻辑

**Martin Fowler**：完全同意。把"语义理解"和"状态管理"分开是一条清晰的界线。LLM 擅长处理模糊的、语义相关的问题；状态管理应该用传统软件工程的可靠机制来处理。

---

## 第五轮：扩展性与演进

**Martin Fowler**：设计文档提到了四个场景的演进路径：

- 场景A：一对多实体展开（本次实现）
- 场景B：话题延伸处理（预留扩展）
- 场景C：条件跳过（预留扩展）
- 场景D：情绪响应（Story 2.3）

这个演进设计是合理的，但我有一个建议：**在每个场景实现后，回顾和重构框架层**。

现在"二分类决策"框架是以场景A为中心设计的。当实现场景B或C时，可能发现需要三分类或四分类决策。这时候应该回到框架层，重新审视抽象是否合理，而不是简单地往里加条件分支。

**吴恩达**：从 AI 产品迭代的角度，我补充一点——**决策边界的清晰性**。

目前的二分类：`NO_CHANGE` vs `REPLAN_ACTION_QUEUE`，边界相对清晰。但扩展到场景B（话题延伸）、场景D（情绪响应）时，决策类别会增加。需要确保每个类别的触发条件是互斥的，避免模型在不同场景之间"犹豫不决"。

---

## 第六轮：可观测性与调试

**Martin Fowler**：第6节提到的可观测性需求很棒：

> 1. 调整记录：每次队列变化记录调整内容、原因、插入位置
> 2. 对象状态标记：记录每个对象的执行状态
> 3. 问题诊断：帮助定位"重复展开"或"未展开"问题

这些都是生产级系统的必要特性。我建议补充一点：**因果链追踪**。

当出现异常（比如某个对象被跳过了）时，需要能够回溯：是哪次决策导致的？那个决策的输入是什么？reason 字段的分析是什么？这样可以帮助脚本开发者理解系统的行为，也便于调试提示词模板。

**吴恩达**：非常同意。在 LLM 应用中，**可解释性**是一个核心挑战。`reason` 字段是一个好的开始，但还可以做得更多：

1. **决策日志**：记录每次意识层调用的完整输入输出
2. **置信度**：让模型输出决策的置信度，低置信度的决策可以触发人工审核或二次确认
3. **版本标记**：标记使用的提示词模板版本，便于追踪模板修改的影响

## 总结：设计评价

### 吴恩达的评价

| 维度         | 评分  | 说明                                                         |
| ------------ | ----- | ------------------------------------------------------------ |
| LLM 应用模式 | ★★★★☆ | 意识层作为决策组件的定位清晰，但提示词模板可以考虑进一步拆分 |
| 决策设计     | ★★★★☆ | 二分类决策简单可靠，扩展性需要关注边界清晰性                 |
| 可解释性     | ★★★☆☆ | 有基础的调试信息，建议增加置信度和因果链追踪                 |
| 工程可行性   | ★★★★☆ | 场景A的实现路径明确，风险可控                                |

### Martin Fowler 的评价

| 维度     | 评分  | 说明                                         |
| -------- | ----- | -------------------------------------------- |
| 架构层次 | ★★★★★ | 职责划分清晰，符合单一职责原则               |
| 扩展性   | ★★★★☆ | 场景演进路径清晰，建议在扩展时及时重构抽象层 |
| 可维护性 | ★★★★☆ | 变量命名和结构设计清晰，文档充分             |
| 设计模式 | ★★★★☆ | 策略模式和模板方法模式的应用恰当             |

### 共识建议

1. **提示词模板拆分**：决策层和执行层分离，降低推理复杂度
2. **对象管理独立组件**：语义理解（LLM）和状态管理（传统代码）分离
3. **因果链追踪**：在可观测性基础上增加决策追溯能力
4. **扩展时重构**：每实现一个新场景，回顾框架抽象是否需要调整

---

## 最终结论

**吴恩达**：这个设计在 LLM 应用层面是合理的。核心思路——"意识层做决策，不做过深的行动规划"——是一个务实的选择，既控制了 LLM 的调用成本，也降低了出错的风险。建议在实现阶段关注对象管理的语义去重问题，以及提示词模板的迭代优化。

**Martin Fowler**：从架构角度，这个设计体现了良好的软件工程思维。分层清晰、职责明确、扩展路径可预期。唯一需要提醒的是：不要让"框架支持扩展"变成过早抽象。建议先让场景A跑通，根据实际运行情况再提取框架层的共性。
